# 5장. 대규모 데이터 처리 실전 입문

**애플리케이션 개발의 급소**

## **강의 14: 용도특화형 인덱싱**

- 데이터량이 커지면 RDBMS 인덱스만으로는 검색 처리에 한계가 있다.
    
    문자열 검색, 조건 조합 검색 등은 B+Tree 인덱스만으로 대응하기 어렵다.
    
- 하테나 북마크는 DB에서 데이터를 주기적으로 추출해 별도의 인덱스 서버에서 검색 인덱스를 재구성하는 방식을 사용한다.
- 애플리케이션 서버는 검색이 필요할 때 DB가 아니라 **인덱스 서버**에 질의한다. (RPC, Web API 형태)
- 키워드 검색을 위해 `Trie 기반 구조`를 사용한다. 문자열의 공통 접두어 검색에 유리하다.
- 카테고리 자동 분류는 `Complement Naive Bayes`로 구현되어 있다. 다량의 문서에서 특정 카테고리로 분류할 때 사용되는 전통적인 통계적 분류 방법이다.
- 이러한 방식은 검색·분류 등 특정 기능을 RDBMS 외부의 전용 시스템에서 처리함으로써 RDBMS는 저장과 기본 조회에만 집중하도록 설계된다.

## **강의 15: 이론과 실전 양쪽과의 싸움**

- 대규모 웹 애플리케이션에서는 이론적으로 ‘이렇게 하는 것이 맞다’는 방식이 실제 환경과 부딪히는 경우가 많다.
- JOIN을 피하라는 실전적인 이유도 파티셔닝된 환경에서 테이블이 서로 다른 서버에 저장될 수 있기 때문이다.
    
    MySQL은 분산 JOIN을 지원하지 않기 때문에 JOIN을 전제로 한 설계가 성립하지 않는다.
    
- 따라서 설계 단계에서는 가능한 JOIN을 사용하지 않고 where in 등의 형태로 쿼리를 나누어 수행하는 방식으로 구조를 잡는다.
- 프레임워크나 기술 선택은 이론만으로 판단하기 어렵다.
    
    데이터 크기, 요청 패턴, 서비스 특성에 따라 실제 적용 가능성은 달라진다.
    
- 현장에서 효과가 있는 설계는 기본 알고리즘과 자료구조 이해를 바탕으로 필요한 곳에 필요한 구조를 배치하는 방식으로 이루어진다.
